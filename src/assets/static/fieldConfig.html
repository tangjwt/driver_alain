字段配置

通过配置的字段，可以解析生成各种请求，例如：JSON,XML,PB等。

字段有两种方式配置：

1、不在配置文件当中配置，直接在case当中指定，

2、在配置文件当中配置，编写case时直接使用对应的name，

假设需要生成以下请求：

xml：

 
<root name="field">
  <sub>
    <subelement>
      <leafs>test1</leafs>
    </subelement>
    <subelement>
      <leafs>test2</leafs>
    </subelement>
    <withDefaultValue>
      <value>values</value>
      <type>abc</type>
    </withDefaultValue>    
  </sub>
</root>

对应的JSON：

 
{
  "root" : {
  "@name" : "field",
    "sub" : {
      "subelement" : [ {
        "leafs" : "test1"
      }, {
        "leafs" : "test2"
      } ],
      "withDefaultValue" : {
        "value" : "values",
        "type" : "abc"
      }
    }
  }
}

其中，若root.withDefaultValue存在，则.type对应的默认值为abc， root有默认属性值为field.

通用规则

以@开头，表示是的xml中的属性， [0]、[1] 代表元素的位置 ，从0开始，[*]代表每次都添加一个新元素。

字段配置时，支持简单的过滤，例如: __root.sub.subelement[leafs='test1'].leafs2__ = "test1" 过滤包含字段leafs且值为test1的subelement，当无法找到匹配项时，会根据过滤条件新建满足要求的subelement。支持属性的过滤，即 subelement[@leafs='test1'] 这种格式，若有多个过滤条件，以逗号分隔，例如：subelement[@leafs='test1',type='testtype']

不在配置文件当中配置

若不想在配置文件当中配置，直接根据case生成请求，那么需要把整个路径都写出来：

 
__root.@name__ = "field"
__root.sub.subelement[0].leafs__ = "test1"
__root.sub.subelement[1].leafs__ = "test2"
__root.sub.subelement[leafs='test1'].leafs__ = "test1"
__root.sub.withDefaultValue.value__ = "values"
__root.sub.withDefaultValue.type__ = "abc"
​

其中，以@开头，表示是的xml中的属性， [0]、[1] 代表元素的位置 ，从0开始。 **注：当使用这种方式时，所有字段的值类型都默认为String类型

当路径中需要包含句号时， 使用 \ 转译，例如，想生成：

 
x
  {  "root" : {  
      "withDefault.value" : {    
        "value" : "values",     
        "type" : "abc"      
      }  
  }

可这么 ：

 
x
__root.withDefault\.value.value__ = "values"
__root.withDefault\.value.type__ = "abc"

注：在配置文件当中配置时，不需要进行转译操作**

注：当请求类型为custom时，不需要进行转译操作，因为custom类型的请求，请求生成由自己控制，不会进行预处理

在配置文件当中配置

当在配置文件当中进行配置时，可以按以下方式编写：

 
​
    <property>
        <description>默认值，请求中不填写也会按默认值生成，若请求中声明，会使用请求中的值覆盖默认值</description>
        <name>name_property</name>
        <parent>root</parent>
        <field>@name</field>
        <type>String</type>
        <default-value>field</default-value>
    </property>
    
​
    <property>
        <description>多子节点，[*] 代表每次都添加一个新的数组元素，若使用数字，例如：[0] ，则只会往数字指定的位置赋值，若已有值，则会覆盖原值</description>
        <allowable-values>string</allowable-values>
        <name>subelement</name>
        <parent>root/sub[*]</parent>
        <field>subelement</field>
        <type>String</type>
    </property>
    
    
    <property>
        <description>只在name属性=abc，且子元素subLeaf的值=bcd 的sub元素中添加subelement，若找不到，会新建一个@name=abc,subLeaf=bcd的sub元素</description>
        <allowable-values>string</allowable-values>
        <name>subelement</name>
        <parent>root/sub[@name='abc',subLeaf='bcd']</parent>
        <field>subelement</field>
        <type>String</type>
    </property>
​
    <property>
        <description>子字段，当在请求中指定default_value字段时，会同时根据配置生成type字段</description>
        <name>default_value</name>
        <parent>root/sub/withDefaultValue</parent>
        <field>value</field>
        <type>String</type>
        <linkproperty>
            <field>type</field>
            <type>enum</type>
            <value>abc</value>
        </linkproperty>
    </property>

此时，若要生成示例中的请求，只需要这么写：

 
__subelemnet__ = "test1"
__subelemnet__ = "test2"
__default_value__ = "values"

对比可以发现，使用配置文件的方式，可减少很多工作量，同时可以对字段添加别名， 当字段含义比较模糊时，使用别名比较直观，也便于理解，另外使用配置文件的方式时，啄木鸟会根据配置的内容自动生成帮助文档，当要添加新人时，能更方便的上手了解请求。

linkproperty

linkproperty 主要用于 当A字段存在，那么B字段的值必定为C 这种情况。

subproperty

subproperty 主要用于 A字段可重复，且A字段包含子字段的情况。 例如，有以下字段 search_substring_descriptor ：

 
​
   message SearchSubstringDescriptor {
      enum SearchSubstringType {
         PLACE = 0;    
         BUSINESS = 1; 
         CATEGORY = 2; 
         IGNORE = 3;   
         ATTRIBUTE = 4; 
               }
      required SearchSubstringType string_type = 1;
      repeat int32 begin_index = 2;
      required int32 end_index = 3;
  }
  repeated SearchSubstringDescriptor search_substring_descriptor = 10;
​

search_substring_descriptor 的可能值为：

 
​
   search_substring_descriptor {
        string_type: PLACE
        begin_index: 0
        begin_index: 1
        end_index: 5
   }
   search_substring_descriptor {
        string_type: BUSINESS
        begin_index: 5
        end_index: 10
   }
​

在这种情况下，可使用subproperty进行配置，填值的时候，子字段的值使用 ‘;’ 分隔，可重复子字段的值使用 ',' 分隔， 当填写的值小于subproperty个数时，按值个数生成subproperty，也可为subproperty填写默认值。若值当中包含 , 或者 ; 时，需要把值用 ' 或者 " 包围起来

 
​
<property>
    <name>search_substring_descriptor</name>
    <parent>ParentPath</parent>
    <field>search_substring_descriptor</field>
    <subproperty>
        <field>string_type</field>
        <type>enum</type>
    </subproperty>
    <subproperty>
        <field>begin_index[*]</field>
        <type>int</type>
    </subproperty>
    <subproperty>
        <field>end_index</field>
        <type>int</type>
    </subproperty>
</property>
​

使用以下请求，可生成上面search_substring_descriptor的可能值：

 
__search_substring_descriptor__ ='PLACE;0,1;5'
__search_substring_descriptor__ ='BUSINESS;5;10'

当填写的值小于subproperty个数时，会按照顺序赋值。

 
__search_substring_descriptor__ ='PLACE;0'

会生成:

 
​
   search_substring_descriptor {
        string_type: PLACE
        begin_index: 0
   }
​

同时，subproperty支持默认值，同样上面的请求，若配置为：

 
​
<property>
    <name>search_substring_descriptor</name>
    <parent>ParentPath</parent>
    <field>search_substring_descriptor</field>
    <subproperty>
        <field>string_type</field>
        <type>enum</type>
    </subproperty>
    <subproperty>
        <field>begin_index</field>
        <type>int</type>
    </subproperty>
    <subproperty>
        <field>end_index</field>
        <type>int</type>
        <default-value>9</default-value>
    </subproperty>
</property>
​

会生成：

 
​
   search_substring_descriptor {
        string_type: PLACE
        begin_index: 0
        end_index: 9
   }
​

可传入值覆盖默认值：

 
__search_substring_descriptor__ ='PLACE;0;10'

会生成：

 
​
   search_substring_descriptor {
        string_type: PLACE
        begin_index: 0
        end_index: 10
   }
​

若要跳过不设置中间的某一个值 ，则直接留空。 空格会认为是有效值：

 
__search_substring_descriptor__ ='PLACE;;10'

会生成：

 
​
   search_substring_descriptor {
        string_type: PLACE
        end_index: 10
   }
​

protocolbuffer

protocolbuffer同样支持上述两种方式，有一个注意点，需要在路径最前面加一个根节点。

比如，有以下protocolbuffer请求：

 
routeAttributes {
  mainTransportType: AUTOMOBILE
}
mainTransportTypeMaxRouteCount: 1
​

直接使用的时候，路径应当分别为：

DirectionRequest.routeAttributes.mainTransportType

DirectionRequest.mainTransportTypeMaxRouteCount

配置的时候，parent应当为：

DirectionRequest/routeAttributes DirectionRequest/mainTransportTypeMaxRouteCount

其中，DirectionRequest代表根节点， 可为任意名称但必须保证一致，建议使用服务request的名称，便于理解。

protocolbuffer extension

假设有以下扩展字段：

 
[aire.ETATrafficUpdateRequestExtension.trafficSnapshot] {
  incidents {
    type: ROAD_CLOSURE
    geoid: 123
    latitude: 39.87297534
    longitude: 116.33212592
  }
}

可使用以下方式进行配置，将扩展字段加入到请求当中（必须提前把扩展字段的相关 jar包等引入项目当中，并配置好）

 
__ETATrafficUpdateRequest.extension.@type__ = "aire.ETATrafficUpdateRequestExtension.trafficSnapshot" 
__ETATrafficUpdateRequest.extension.incidents[0].type__ = "ROAD_CLOSURE"
__ETATrafficUpdateRequest.extension.incidents[0].geoid__ = "123"
__ETATrafficUpdateRequest.extension.incidents[0].latitude__ = "39.87297534"
__ETATrafficUpdateRequest.extension.incidents[0].longitude__ = "116.33212592"

基中ETATrafficUpdateRequest代表服务名，需要与配置文件、其它请求字段一致，

extension是固定的关键字，它的属性type的值为扩展名

其它就是具体的赋值，

若是一个请求中有多个extension，只需要给extension加一个下标就可以了，例如：

 
[aire.need_latency]: true
[aire.trafficSnapshot] {
  incidents {
    geoid: 123
  }
}

对应的写法如下：

 
__ETATrafficUpdateRequest.extension[0].@type__ = "aire.need_latency" 
__ETATrafficUpdateRequest.extension[0]__ = "true"
__ETATrafficUpdateRequest.extension[1].@type__ = "aire.trafficSnapshot"
__ETATrafficUpdateRequest.extension[1].incidents.geoid__ = "123"
